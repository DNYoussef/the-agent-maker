# Phase 1-4: Complete UI/UX Specification

**Version**: 2.0
**Date**: 2025-10-16
**Purpose**: Complete user interface specification for Phases 1-4 with adaptive controls and real-time visualization

---

## Executive Summary

This document specifies the **complete UI/UX system** for Agent Forge V2 Phases 1-4. The UI must provide:
1. **Adaptive controls** that adjust to runtime model sizes
2. **Real-time monitoring** with failure detection visualization
3. **Interactive intervention** when critical issues arise
4. **Progressive disclosure** (simple by default, advanced on demand)
5. **Visual feedback** for all 12 critical failure modes

### Design Philosophy

- **Local-First**: All UI runs on local machine, no cloud dependencies
- **Real-Time**: Updates via W&B webhooks or local WebSocket
- **Adaptive**: Controls change based on detected model size
- **Fail-Safe**: User can intervene at any checkpoint
- **Transparent**: Show what the system is doing and why

---

## Technology Stack

### Frontend
- **Framework**: Next.js 14 + React 18 + TypeScript
- **State Management**: Zustand (client) + TanStack Query (server)
- **Visualization**:
  - Recharts (metrics, loss curves)
  - D3.js (evolution trees, merge graphs)
  - Three.js (3D model architecture viz)
  - Plotly (real-time streaming charts)
- **UI Components**: shadcn/ui + Tailwind CSS
- **Real-time**: Socket.IO or native WebSocket

### Backend Integration
- **Local API**: FastAPI (Python) on localhost:8000
- **Weights & Biases**: Local W&B server or cloud sync
- **Data Flow**: Python â†’ W&B â†’ WebSocket â†’ React UI

## API Endpoints Required

### Phase 1
- `POST /api/phases/1/configure` - Set 3-model configs
- `POST /api/phases/1/start` - Start Phase 1 training
- `GET /api/phases/1/status` - Real-time status
- `POST /api/phases/1/pause` - Pause training
- `POST /api/phases/1/resume` - Resume training
- `POST /api/phases/1/checkpoint` - Manual checkpoint
- `GET /api/phases/1/diversity` - Diversity metrics

### Phase 2
- `POST /api/phases/2/configure` - Set evolution params
- `POST /api/phases/2/start` - Start evolution
- `GET /api/phases/2/status` - Generation status
- `GET /api/phases/2/population` - Current 8 models
- `GET /api/phases/2/diversity` - Diversity score
- `POST /api/phases/2/intervene` - Diversity boost

### Phase 3
- `POST /api/phases/3/generate-data` - Start data generation
- `GET /api/phases/3/generation-status` - Data gen progress
- `POST /api/phases/3/step1/start` - Start baking
- `GET /api/phases/3/step1/status` - Baking status
- `POST /api/phases/3/step2/start` - Start RL
- `GET /api/phases/3/step2/status` - RL status
- `POST /api/phases/3/anti-theater` - Run anti-theater check

### Phase 4
- `POST /api/phases/4/configure` - Set compression params
- `POST /api/phases/4/start` - Start compression
- `GET /api/phases/4/status` - Layer-by-layer status
- `GET /api/phases/4/sparsity` - Sparsity heatmap data
- `POST /api/phases/4/revert-layer` - Revert layer quantization

### Global
- `GET /api/pipeline/status` - Overall pipeline status
- `POST /api/pipeline/pause` - Pause entire pipeline
- `POST /api/pipeline/resume` - Resume pipeline
- `GET /api/hardware/status` - GPU, VRAM, RAM, Temp
- `GET /api/wandb/runs` - List W&B runs
- `GET /api/settings` - Get all settings
- `POST /api/settings` - Update settings

### WebSocket Events
- `phase:progress` - Phase progress updates
- `phase:metric` - New metric logged
- `phase:alert` - Failure detection alert
- `phase:complete` - Phase finished
- `hardware:update` - GPU/VRAM/RAM/Temp update
